# Deuda Técnica – SIGHA API

Esta tabla resume los problemas técnicos detectados en el código fuente, junto con su posible impacto y propuestas de solución.

| Categoría | Descripción | Impacto / Riesgo | Solución Propuesta |
|-----------|-------------|------------------|--------------------|
| **Controladores gordos** | Los controladores (`*.controller.ts`) contienen gran cantidad de endpoints y lógica de autorización directamente en los métodos, lo que dificulta su mantenimiento y pruebas unitarias. | Alto: dificultad para añadir nuevas rutas sin introducir errores, bajo coverage de pruebas. | Separar la lógica de negocio en servicios dedicados, mantener los controladores como capa de orquestación mínima. |
| **Falta de pruebas unitarias** | Sólo existen dos archivos de pruebas (`identification-format.validator.spec.ts` y `logger-sanitizer.spec.ts`). No hay tests para la mayoría de los controladores, servicios ni DTOs. | Alto: riesgo de regresiones en cambios de negocio; cobertura de código muy baja. | Introducir pruebas unitarias y e2e para cada controlador y servicio usando `jest` y `supertest`. |
| **Dependencias potencialmente obsoletas** | Algunas dependencias en `package.json` están en versiones antiguas (ej. `bcrypt@^6.0.0`, `@nestjs/common@^11.1.6` aunque reciente, `winston-daily-rotate-file@^5.0.0`). No se ha verificado si existen versiones menores de seguridad. | Medio: posibles vulnerabilidades no parcheadas. | Ejecutar `npm outdated` y actualizar a versiones mayores que incluyan correcciones de seguridad, revisando notas de migración. |
| **Hardcodeo de mensajes de error y strings** | Mensajes de error y descripciones se encuentran duplicados en varios controladores (p. ej., "Sin permisos suficientes"). | Bajo: aumenta la superficie de cambios futuros y posibles inconsistencias. | Centralizar textos de error en un módulo de constantes o usar `i18n` para facilitar mantenimiento. |
| **Variables de entorno sin validación** | El código asume la presencia de variables como `JWT_SECRET`, `RESEND_API_KEY`, etc., sin validar su existencia ni su formato al iniciar la aplicación. | Medio: posibles fallos en tiempo de ejecución si faltan valores críticos. | Añadir una capa de validación (p. ej., `class-validator` con DTO de configuración) al cargar `.env`. |
| **Falta de paginación consistente** | Algunos endpoints devuelven listas sin parámetros de paginación (`findAll` en varios controladores) mientras que otros sí lo hacen. | Bajo: pueden generar respuestas muy grandes y afectar rendimiento. | Uniformizar la API introduciendo parámetros `page` y `limit` en todos los listados. |
| **Uso de `any` y tipos implícitos** | En varios lugares (p. ej., en los controladores se usan tipos implícitos en respuestas o parámetros) se omite la definición estricta de tipos TypeScript. | Bajo: reduce la capacidad de detección de errores en tiempo de compilación. | Habilitar `noImplicitAny` y definir interfaces/DTOs para todas las respuestas y argumentos. |
| **Duplicación de lógica de autorización** | Cada controlador repite verificaciones de roles (`@Roles(...)`) y condiciones de 2FA. | Medio: mayor probabilidad de inconsistencias y errores de autorización. | Implementar guardia global o interceptor que centralice la lógica de autorización basada en metadatos. |
| **Registro de auditoría parcial** | El logger sanitiza PII/PHI pero no se asegura que todos los eventos críticos (p. ej., cambios de estado de usuarios) se registren en el nivel `audit`. | Medio: falta de trazabilidad completa para auditorías regulatorias. | Revisar todos los servicios críticos y agregar llamadas a `logger.audit` con la información necesaria. |

| **Notificaciones - Push en tiempo real** | Las notificaciones solo se obtienen mediante peticiones HTTP; no hay mecanismo push (WebSocket/SSE). | Medio: experiencia de usuario pobre, latencia en la entrega, carga innecesaria al cliente. | Implementar un gateway NestJS con WebSocket o Server‑Sent Events para push de notificaciones; usar un broker (Redis) para escalar. |
| **Cache de notificaciones** | Se usa la clave `notifications-unread::*` y se elimina con wildcard, afectando a todos los usuarios. | Bajo: sobre‑borrado de caché, posible pérdida de datos en alta concurrencia. | Refactorizar a caché por usuario (`notifications-unread:${userId}`) y establecer TTL; usar patrones de invalidación más finos. |
| **Validación de archivos adjuntos** | No se valida el tipo MIME ni la extensión de los archivos adjuntos en `CreateNotificationDto`. | Medio: riesgo de subir archivos peligrosos o incompatibles. | Añadir validaciones de MIME y extensión en DTO y servicio; limitar tipos permitidos (pdf, jpg, png, etc.). |
| **Manejo de concurrencia al marcar como leída** | La actualización del estado de la notificación no está envuelta en transacción ni usa control de versiones. | Bajo: posible condición de carrera si varios procesos marcan simultáneamente. | Utilizar transacciones de TypeORM o `optimistic locking` con columna de versión para asegurar consistencia. |
| **Job de envío programado** | No existe proceso que envíe automáticamente notificaciones pendientes (`nSent = false`). | Medio: notificaciones pueden quedar sin enviar, requerir intervención manual. | Implementar un job (Bull, Agenda o worker thread) que procese notificaciones pendientes, actualice `nSent` y maneje reintentos. |

> **Nota**: Cada ítem debe priorizarse en el backlog de desarrollo y asignarse a un sprint de refactorización. Se recomienda crear tareas en el sistema de gestión de proyectos con la información detallada de esta tabla.
